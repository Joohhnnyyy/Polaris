What you are building
CivicMind is a multi-agent urban intelligence platform. The core idea: a single civic complaint is noise — a cluster of 3 correlated complaints in 300m over 72 hours is a signal of a deeper infrastructure failure. The system detects that signal automatically, reasons about the cause, and drafts the action for a human to approve.
Next.js 14 PWA FastAPI (Python) Supabase (PostgreSQL + pgvector) Gemini 2.5 Pro Google Maps JS Redis Vercel + Cloud Run
The 3 interfaces you are building
1
Citizen PWA mobile-first
A simple form: photo upload + GPS location + short description. The citizen submits. That is all they do. The system does the rest. This is the demo entry point — the judge scans a QR code and submits a report live.
2
Officer Copilot Dashboard demo centerpiece
A desktop dashboard showing active clusters, risk briefs, the Evidence Web graph, and draft emails to ward engineers. This is where Urban Autopsy Mode runs visibly. The judge watches this screen during the demo.
3
Public Heatmap read-only
A Google Maps view showing risk clusters color-coded by severity, plus predicted hotspots. No login required. Shows the city's live status to anyone.
The signature feature: Urban Autopsy Mode
When a new report arrives, this runs automatically. The judge watches it happen in real time. This is what wins the hackathon.
1
Intake Agent
Gemini Vision classifies the photo. Extracts: issue type, severity 1–5, area affected, surface type.
2
Evidence Agent
Deep Gemini Vision forensics. Queries all issues within 300m from the past 90 days. Pulls infrastructure maintenance history for the zone.
3
Synthesis Agent ← most important
Gemini 2.5 Pro in a multi-turn function-calling loop. Calls tools, reasons about causal chains, assigns confidence scores, produces the Risk Brief JSON.

Takes the Risk Brief. Finds the responsible ward officer. Generates a professional draft email. One click to send.

Day 1Foundations — get a report into the database
What you are building today
A citizen submits a photo + location → Gemini Vision classifies it → it is saved to PostgreSQL. That's the full loop for day 1. Nothing else.
Input
📷 Photo (JPEG/PNG, up to 5MB)
📍 GPS coordinates (lat, lng)
📝 Short description (optional)
Output
✅ Issue saved to DB with UUID
🏷️ Category (pothole / leak / etc)
⚠️ Severity score (1–5)
📊 Gemini analysis JSON
Step-by-step for day 1
1
Create Supabase tables
In Supabase SQL editor, run: CREATE EXTENSION vector; then create the issues table with columns: id, category, severity, lat, lng, description, images (text[]), status, embedding (vector(768)), gemini_analysis (jsonb), created_at.
2
FastAPI: POST /reports endpoint
Accepts multipart form: image file + lat + lng + description. Saves image to Supabase Storage. Returns issue_id. This is the backend entry point.
3
Intake Agent: call Gemini Vision
After saving the image, call Gemini 2.5 Pro with the image + this system prompt: "Classify this civic issue. Return JSON: {issue_type, severity (1-5), area_affected_m2, infrastructure_type, confidence}". Save the JSON response to gemini_analysis column.
4
Frontend: citizen report form
Next.js page at /citizen. Three fields: camera/file input, a "Use my location" button (browser Geolocation API), text area. Submit button POSTs to your FastAPI. Show a success screen with the issue_id.
5
Verify end-to-end
Submit a real photo of a road. Check Supabase table — the row should appear with the Gemini classification JSON filled in. If this works, day 1 is done.

📍 Coordinates (lat, lng)

Step-by-step for day 2
1
Evidence Agent: deep Gemini Vision call
Different prompt from Day 1's classification. This one says: "You are an infrastructure forensics expert. Describe: physical dimensions estimate, likely root cause, infrastructure systems affected, age of damage, risk of cascading failure. Be technical." Save as evidence_analysis in the issues table.
2
Generate and store embeddings
Use models/text-embedding-004 via the Gemini API on the issue description + category text. Store the 768-dim vector in the embedding column. This enables semantic similarity search later.
3
pgvector similarity search function
Write a Supabase SQL function: find_similar_issues(embedding vector, radius_m float, days_back int) that returns all issues within X meters and Y days ordered by cosine similarity. This is used by the Synthesis Agent tomorrow.
4
Seed 50 demo issues — most important task of day 2
Write seed_demo_data.py. Create 3 zones in Ghaziabad. Zone A (Sector 7B): 4 issues over 5 days — 2 potholes, 1 water leak, 1 pavement subsidence — all within 300m. This is the cluster that Urban Autopsy Mode will detect during the demo. Zone B: 6 unrelated scattered issues (negative space). Zone C: 3 older issues from 6 months ago showing the historical pattern. Hardcode the lat/lngs to real Ghaziabad coordinates.
Day 3 ⚡ CriticalSynthesis Agent — the heart of the system
If Day 3 slips, cut features. Do not skip this.
The Synthesis Agent is the only part of CivicMind that no other team will build. Everything else — the UI, the heatmap, the email drafts — is dressing. This is the actual agentic intelligence. Give it the full day.
What the Synthesis Agent does
It runs Gemini 2.5 Pro in a multi-turn function-calling loop. Gemini decides which tools to call, calls them, sees the results, calls more tools, and eventually produces a causal hypothesis. Your code does not decide what to look for — Gemini does.
Input
🆕 Newly submitted issue (trigger)
📍 Its coordinates
🗃️ Access to all tools below
Output (Risk Brief JSON)
🔴 risk_level: CRITICAL/HIGH/MEDIUM
💡 causal_hypothesis (text)
🔗 evidence_chain (array of issues)
👥 affected_residents (int)
💰 cost_estimate (₹)
📊 confidence (0.0–1.0)
The 4 tools you must implement for the function-calling loop
1
query_nearby_issues(lat, lng, radius_meters, days_back, categories)
Queries your PostgreSQL. Returns all issues within the radius and time window. Gemini calls this first to gather the cluster. This is a simple SQL query — 20 lines of Python.
2
fetch_infrastructure_history(zone_id, years_back)
Returns maintenance records from your seeded historical data. For demo, this is a JSON file you create — real city data would come from a municipal API. Include water main age, last repair date, material type.
3
assess_causal_chain(issue_ids, hypothesis)
This is a recursive Gemini call inside the loop. Feeds the listed issues + hypothesis back into Gemini and asks it to score the confidence. Returns a float 0–1.
4
generate_risk_brief(cluster_data)
Takes everything Gemini has gathered. Makes a final Gemini call with a structured output schema to produce the Risk Brief JSON. This is the terminal step of the loop.
The exact Synthesis Agent system prompt
You are CivicMind's Urban Intelligence Agent for Ghaziabad. A new civic issue has been reported. Your task: 1. Call query_nearby_issues to find related reports in the area 2. Call fetch_infrastructure_history for the zone 3. Identify causal relationships between the issues 4. Generate 1-3 hypotheses about the root cause 5. Assign confidence scores using assess_causal_chain 6. Call generate_risk_brief with your findings Rules: - Use tools before concluding. Do not guess without data. - If no cluster exists (fewer than 2 related issues), output risk_level: LOW - Be specific: name the infrastructure (e.g. "water main on NH58") - Estimate affected_residents from the 500m population density

What you are building today
The full frontend. Every screen. Do not polish — get all the data flowing. Polish is Day 6. Use shadcn/ui components for speed. The demo path screens come first.
Screen 1: Officer Copilot Dashboard (most important)
A
Active clusters panel (left sidebar)
List of clusters ordered by risk_level. Each row shows: red/amber/green dot + zone name + issue count + confidence %. Clicking a row loads the full Risk Brief.
B
Risk Brief card (center)
Shows: risk level badge, causal_hypothesis text in large readable font, evidence_chain list with each connected issue, affected_residents count, cost comparison (preventive vs reactive), confidence score. This is what judges read during the demo.
C
Agent execution trace (right panel)
Real-time log of what the Synthesis Agent did. Shows each tool call as it happened: → query_nearby_issues (300m, 90 days) → Found 3 issues → fetch_infrastructure_history → Water main age: 47yr → confidence: 0.73. Stream this via WebSocket. This is the "watching the city think" moment.
D
Draft email + "Send to Officer" button
Show the Gemini-generated email below the brief. Big "Review and Send" button. On click, show a confirmation modal. This is the one-click autonomous action demo moment.
Screen 2: Evidence Web (D3.js force graph)
A force-directed graph where each issue is a node, and edges represent causal connections weighted by confidence. Node color = issue type. Edge thickness = confidence score.
1
Install D3.js in your Next.js project
npm install d3. Create a React component EvidenceWeb.tsx that takes a cluster's evidence_chain array as props and renders the graph. Nodes are clickable — clicking shows the issue's photo and Gemini analysis.
Screen 3: Google Maps heatmap
Use the Google Maps JavaScript API's HeatmapLayer. Fetch all cluster center coordinates + risk scores from your API. Map risk_level to color: CRITICAL = red, HIGH = amber, MEDIUM = yellow. Add a toggle to show predicted hotspots as pulsing markers.
